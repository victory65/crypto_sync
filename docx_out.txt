CRYPTO SYNC — Backend & Trade Mirroring System
Overview
Crypto Sync is a mobile-only real-time trade mirroring app that allows users to connect multiple crypto exchange accounts. Trades executed on a Master account are instantly mirrored across Slave accounts, with lot size customization and full failure transparency.
This write-up focuses on backend architecture, security, logging, and scalability — based on the interactive session.
1. Backend Architecture
Two-Service Separation
We designed the backend with two separate Python services for security, scalability, and clarity:
A. User DB Service
Stores:
Users, Master & Slave accounts
Subscription plans
Positions and historical trade execution logs
API keys encrypted using AES-256 with server master key
Handles mobile UI requests (login, account management, trades)
Communicates with Trade Service via ZeroMQ REQ/REP + PUB/SUB
Maintains logs of critical events only
B. Trade Service
Executes trades via CCXT library
Mirrors Master trades across Slave accounts in real-time
Applies lot sizes per slave (Fixed or % of balance)
Handles parallel retries for failed trades (max 3 attempts)
Returns execution outcomes to DB service
Logs critical trade attempts, failures, and retries
Benefits of this separation:
Clear division of responsibility
Trade execution is isolated from user data storage
Easier to migrate services independently if scaling or server change is required
2. Trade Execution Flow
Trade Initiation
User places trade on Master account via mobile UI
DB Service validates user, subscription, and account limits
API Key Decryption
DB Service decrypts Master + Slave keys in memory
Sends securely to Trade Service via ZeroMQ
Lot Size Application
Each Slave account applies Default Lot Size:
Fixed: e.g., 0.05 BTC
Percentage: e.g., 15% of balance
If insufficient funds, marked for retry
Trade Mirroring
Trade Service executes trades asynchronously across all slaves
Each slave’s execution status updated: Filled, Partial Fill, Rejected, API Error
Failure Handling & Parallel Retry
Failed trades retried in parallel up to 3 times
Retry delay configurable (1–2 seconds)
Final status returned to DB Service
UI Update & Transparency
Mobile UI receives real-time updates via PUB/SUB
Position Detail screen shows per-slave status, lot size, and error reasons
Logging
Critical events logged: master trade, slave execution status, retries, errors
Logs rotated to maintain manageable file sizes
Historical logs retained 12 months, then purged
